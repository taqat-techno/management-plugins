# Global Lessons Learned — Cross-Project Best Practices

> Accumulated from real project sessions. Apply these across ALL projects.
> Last updated: April 21, 2026 (482 lessons, 83 categories)
>

## Report Writing

1. **Be specific, not vague** — "Udemy training completed" > "Core training completed". Name the platform, tool, or system explicitly.
2. **Reports must stand alone** — The reader (CEO, Operation Manager, Engineering Manager) shouldn't need the author in the room to explain what a cell means.
3. **Every cell should be self-explanatory** — If someone reads a table cell and asks "what does this mean?", rewrite it.
4. **Same state = same label** — Use identical wording across all rows/departments for the same status. Don't say "Phase completed" for one and "Training done" for another.
186. **Pearl Pixels Quarterly Report and Q1 Report are complementary, not duplicates** — The quarterly report contains unique PM observations (execution-vs-planning mode, untracked study hours, QA validation gaps, zero-activity team members) not in the main Q1 report. Always read both when building Q2 plans or executive summaries.
187. **"Split" means separate into individual entries, never remove** — When user says "split" two items combined in one row, create separate entries for each. "Split" preserves all items, just separates them. Never interpret it as removing the second item.
227. **Verify team roles from the authoritative team directory before assigning action item owners** — Never assume roles from email CC lists or job title signatures. RC Hosting Investigation assigned backend tasks to an Implementation team member (El-afify) and an unlisted person (Badawy) based on email CC assumptions instead of checking DevOps_Teams.md. Always cross-reference the team directory first.
253. **Deep-study all repos and wikis before creating a dashboard** — A shallow pull-and-list misses wiki content (SOPs, architecture decisions, project plans, test reports). Clone wikis, read every file, and explore all branches before claiming a "complete" report. The Taqat dashboard missed 35 wiki pages (5,461 lines) on first pass.

## Status Labels & Completion Markers

5. **"Ongoing" is a lazy status** — Replace with specific next steps: "Phase 2: To Be Defined", "Certification prep continues independently (lab practice & exam)". Show intentional planning, not vagueness.
6. **Completion markers must be consistent** — If one department shows "Training completed" in a phase, ALL completed departments must show it. Never leave blank or `--` for a completed item.
7. **Explain what "independently" means** — If something continues outside the plan scope, say what specifically continues (e.g., "lab practice & exam"), not just "continues independently".

## Bilingual Documents (EN/AR)

8. **Always update both languages for every text change** — Never edit English without updating the Arabic span. Easy to forget on small fixes.
9. **Verify RTL visually** — Arabic layout can break borders, alignment, and table padding. Always open the HTML and toggle to Arabic after changes.
10. **Check both languages after toggle** — A change that looks fine in English might overflow or misalign in Arabic RTL mode.

## Stakeholder Communication & Emails

11. **Give email options, not just one draft** — Provide 3 versions: formal (detailed), concise (professional), action-oriented (direct). Different audiences need different tones.
12. **"Dear All" for multiple managers** — Avoids hierarchy issues. Let the To/CC fields handle the addressing.
13. **Always include what's attached and why** — Don't just say "please find attached". List what the attachment contains (bullet points).
282. **Record email directives into deliverables immediately** — Semir's Mar 16 email changed Flow 2 ownership (Mohammed Agha → Ashik Mohamed/TaqaTechno) but sat unrecorded for 3 weeks, leaving the V2 summary stale with wrong RACI. Always update execution documents when new email directives arrive.
283. **Always verify and use correct job titles in formal documents** — "IT Project Manager" vs "Project Manager & Scrum Master" is a small detail but a big difference in formal deliverables. Cross-check titles against the latest org chart or email signatures before publishing.

## HTML Deliverables

14. **Open HTML after every change** — Code edits can look correct but render wrong. Always verify in the browser.
15. **Version control in filenames** — Use `_v2`, `_v3` suffixes. Keep old versions in versioned folders (`v1/`, `v2/`, `V3/`).
277. **Assessment files for candidates must NOT contain evaluator content** — When cloning assessment tabs into standalone HTML for email, exclude scoring rubrics, reference solutions, interview questions, scenarios, and PM decisions. Candidates see only requirements + deliverables. Evaluator content stays internal. Cost: leaking the scoring rubric lets candidates game the assessment.
284. **Changelog sections are internal tracking, not for stakeholder deliverables** — V2→V3 diff tables help the PM track what changed, but management-facing HTML should not include version changelogs. Remove them from final outputs presented to the board or stakeholders.
285. **Always reference organization identity explicitly in deliverables** — Include the Azure DevOps org name (TaqaTechno) and URL (dev.azure.com/TaqaTechno) in HTML reports. Never leave organization references generic when the audience needs to know which tenant/org is affected.
308. **Hardcoded candidate names make assessment pages non-reusable** — team-assessment-phase0.html had "Shehab & Hala" in 30+ places. When new candidates join, every reference breaks. Use generic terms ("candidates", "current backend team") from the start. Names belong in runtime data (contenteditable fields), not static HTML.
309. **Every HTML deliverable must include a favicon** — Missing favicons make browser tabs show a generic icon, making it hard to identify dashboards when multiple tabs are open. Use inline SVG data URIs (`<link rel="icon" type="image/svg+xml">`) so favicons work without external files. Check all linked HTML files during dashboard reviews.

## Data Validation

16. **Always validate email/source dates first** — Before merging versions, check if source files actually changed. File size comparison catches identical files.
17. **Cost model is the #1 change trigger** — Major version changes are usually driven by pricing/subscription model changes, not content changes.
18. **Use PowerShell COM for .msg extraction** — `Outlook.Application` works but files must not be open in Outlook.
286. **No assumptions on source data — always read the actual files** — Never assume PDF/MD content matches what you remember. The V3 Recommendation had exact user lists (14 users, 2 teams), cost breakdowns (3 scenarios), and 6 role definitions that all had to match the email summary precisely. Always re-read the source before generating derivative documents.
292. **Always verify live system data before finalizing deliverables** — The V3 recommendation (Feb 3) said 24 users. Live Azure DevOps API check (Apr 7) showed 33 users — a 37% growth in 2 months that no document reflected. Always query the live system before publishing user counts, costs, or scope.

## OKR & KPI Dashboards

19. **OKR scoring must be automatic** — Don't rely on manual status entry for KR progress. Use editable "Current" fields that auto-calculate progress percentage, bar color, and status badge. Manual scoring drifts and becomes inconsistent.
20. **Inverse metrics need special handling** — KPIs where lower is better (bug escape rate, carry-over tasks) need inverse progress calculation: if current <= target, it's 100%. Don't use the same formula as "higher is better" metrics.
21. **Separate OKR (strategic) from KPI (operational)** — OKRs are quarterly aspirational goals with key results. KPIs are ongoing team metrics. Mixing them in one table confuses the audience. Use separate tabs/sections.
183. **Dashboard team rosters must stay in sync with actual assignments** — When a team member is added or moved between teams in DevOps, the dashboard roster (HTML arrays, WIQL member lists, team card tables, member counts, and header totals) must ALL be updated together. A member missing from any one array silently drops their metrics from queries. Check all data arrays — dashboards often have 3-4 redundant member lists that must match.

## Search Index Maintenance

22. **Rebuild search index after every pull** — When new HTML pages are added to a repo with a client-side search engine, the search index (e.g., `search-index.js`) won't include them automatically. Always check if new pages need to be added to the file list AND rebuild the index.
23. **New pages must be registered in build script** — Adding a new HTML file to the repo doesn't make it searchable. The file must also be added to the FILES array in the build script (e.g., `build-index.js`). This is a two-step process: (1) add to list, (2) rebuild.
312. **New root-level HTML files are not auto-indexed in the Sessions Index** — Creating a new HTML file in a project folder does not make it searchable in the Claude Sessions Index. A session card must be manually added to both `Claude_Sessions_Index.html` and `.md` with the correct path, badges, and date for the file to appear in search results.

## Project Estimation & Story Points

24. **Always verify project duration across all files** — Different documents (backlog, proposal, basis of calculation) may show conflicting durations. Cross-check all sources after every pull — the latest versioned file (v2) is the source of truth.
25. **Story points need a documented scale** — If a backlog uses SP without defining what each value means, derive the scale from actual story examples. Group stories by SP value, analyze complexity patterns, and document the scale for team alignment.
26. **Person-months ≠ calendar months** — Always calculate effective person-months (allocation % × duration) not headcount × months. A team of 11 at mixed allocations (20%-100%) may only deliver 15.93 person-months in 4 calendar months.
27. **AI acceleration changes all estimates** — When a project uses AI-assisted development, traditional effort estimates don't apply. Document the multiplier per role (QA gets highest ~4x, PM gets lowest ~1.5x) so stakeholders understand the compressed timeline.

## DevOps API Integration

28. **Use real DevOps states, not made-up labels** — Dashboard metrics must match the exact states in Azure DevOps (To Do, In Progress, Resolved, Done, Closed). Using generic terms like "Active" or "Carry-over" creates confusion when comparing with the DevOps board. Always query actual states first with the States API before building queries.
29. **Query by Assigned To when Area Paths don't exist** — If a DevOps project doesn't have team-specific area paths (Backend, Frontend, etc.), query by `[System.AssignedTo] CONTAINS 'member name'` across all projects. This is more flexible than area paths and works cross-project.
30. **Always provide a Data Source tab** — When building dashboards that fetch from APIs, include a transparent "Source" tab showing the exact queries used, results returned, and verification links. The user should never have to trust the numbers blindly — they need to verify against the source system independently.
31. **Discover projects via API before hardcoding** — Don't assume which DevOps projects exist based on email analysis or conversation context. Use `/_apis/projects` to discover all projects, then let the user confirm which ones to include.
32. **Never expose PAT tokens in chat or files** — PAT tokens should only be entered in browser modals and stored in localStorage. Never save them in code, memory files, or conversation. If a user shares one, remind them not to.
33. **Arabic toggle must be planned from day one** — Every PM dashboard needs EN/AR toggle. Adding it after the fact requires touching every text element. Plan for `data-i18n` attributes from the start to avoid rework.
34. **"Last updated" means fetch time, not current time** — Dashboard timestamps should show when data was last retrieved from the API, not a live clock. A live clock gives false confidence that data is current when it may be hours old.
35. **Show full project names, not abbreviations** — In dashboards for CEO/Board, use full project names (e.g., "KhairGate (BMS)") instead of abbreviations (e.g., "BMS"). Abbreviations are internal shorthand that external stakeholders won't recognize.
293. **Azure DevOps API does not store job titles** — The user list API (`az devops user list/show`) only returns display name, email, and license type. Job titles must come from the user, HR system, or Microsoft Entra ID. Don't assume you can pull titles from DevOps.

## Session Workflow & Git Hygiene (Mar 16, 2026)

36. **Always pull before analyzing** — Never analyze or report on data without pulling latest from remote first. Stale local data leads to wrong conclusions (e.g., reporting 9-month plan when v2 already shows 4 months).
37. **URL.txt is the session start ritual** — For repos with a URL.txt, always run it first: fetch → compare → pull if needed → open HTML. This prevents working on outdated files.
38. **Check git diff after pull, not just commit message** — Commit messages like "design modifications" are vague. Always check `git diff --stat` to know exactly which files changed and how much, so you can assess impact.
39. **Version files may have internal inconsistencies** — A v2 file may still contain v1 data in some sections (e.g., team member durations showing 9 months in a 4-month plan). Flag these to the user rather than silently assuming one is correct.
278. **Remove project-specific context from standalone assessment files unless PM explicitly wants it** — When creating assessment HTML for email, candidates need technical requirements — not PM governance context (project names, phase numbers, sprint references, hiring gaps). Only include project context if the PM explicitly confirms it belongs in the candidate-facing file.
306. **Always run git pull first, not just git log** — `git log` shows fetched commits but doesn't update the working tree. Today's session checked `git log` (saw 4 commits) but didn't pull, leaving local 2 commits behind. The drift fix was applied correctly but the repo was stale. Always `git fetch` + `git pull` as the first action.

## Data Analysis & Presentation (Mar 16, 2026)

40. **Derive what's not documented** — If a backlog has story points but no scale definition, reverse-engineer it from the data. Count distribution, group by SP value, read actual stories at each level, then present the derived scale for validation.
41. **Show the math, not just the result** — When presenting person-months, show: Allocation % × Duration = Person-Months for each member. Stakeholders trust calculations they can verify.
42. **Compare before and after** — When two versions of a plan exist (v1 vs v2), always present a variance table. Decision-makers need to see what changed and by how much (e.g., -56% duration, -66% effort).
43. **Define jargon when asked** — Terms like "person-month", "story point", "sprint velocity" are not obvious to all stakeholders. When the user asks "what does X mean?", give a concrete example from their own project data, not a textbook definition.

## Memory & Lessons Discipline (Mar 17, 2026)

44. **Save lessons at end of every session** — Don't wait for the user to ask. At the end of each working session, proactively capture lessons learned into `global_lessons.md` and project-specific memory files.
45. **Memory files are for future conversations, not current tasks** — Don't save ephemeral task details (current debug state, temp file paths). Save things that help future-you understand the project: team composition, estimation methodology, key decisions, file relationships.
46. **Update session log after every pull** — The session log in MEMORY.md is the project timeline. Record: date, commit range, new/updated files, key actions taken. This helps trace when changes were introduced.

## Consolidation & Multi-Source Architecture (Mar 24, 2026)

47. **Never modify source files in consolidation projects** — Keep all source analyses untouched (e.g., S1: 468 files, S2: 21 files). Build an independent consolidated output (HUB v3.0). This preserves original analysis integrity for audit trails.
48. **Track conflicts explicitly with IDs and side-by-side display** — Number conflicts (CF-01 to CF-12) and show both positions without auto-resolution. Let governance decide. Transparent conflict management builds stakeholder trust.
49. **Color-coded source attribution on every data point** — Use consistent color tags (S1=blue, S2=orange, Merged=purple, New=green) so readers can trace any claim to its origin. Critical for multi-source projects.
50. **3-tier gap classification prevents analysis paralysis** — Classify gaps as Blocking / Important / Deferrable. Simple 3-tier system distinguishes urgency without overcomplicating prioritization.

## Portal & Pipeline Architecture (Mar 24, 2026)

51. **Zero-dependency static HTML is the most durable deliverable** — No build system, no bundler, no server. Works offline, on any machine, forever. Essential for board presentations and long-term archival.
52. **Modular CSS from day one saves rework** — Separate CSS files (variables, base, layout, nav, components, RTL, print) allow targeted changes without breaking unrelated pages. Invest in CSS architecture early.
53. **Numbered scripts enforce execution order** — Name pipeline scripts 01→22 to make execution order self-documenting. Anyone can run them in sequence without reading a README.
54. **Large JSON intermediates enable independent downstream scripts** — Files like `db_objects.json` (20MB) are large but let any downstream script work without re-extracting from source. The storage cost is worth the decoupling.
55. **FOLDER_OBJECTIVE_MAP.html serves as the project atlas** — A visual map of all folders and their purposes is essential when projects exceed 10 subfolders. New team members can orient themselves immediately.
56. **Recommendations stay as recommendations until governance approves** — Never auto-apply analysis recommendations to source files. Present findings; let the board decide. Technical teams provide analysis; governance decides.

## Dashboard Maintenance & Adoption (Mar 24, 2026)

57. **Maintenance burden kills dashboards** — If a dashboard requires > 30 manual inputs per week for one person, it won't be used. v1 had ~113 manual inputs and was never populated. v2 cut to ~24 by auto-fetching everything possible from DevOps APIs. Design rule: if an API can provide it, never ask the PM to type it.
58. **KRs must be outcomes, not activities** — "Weekly agendas with DevOps metrics pre-populated" is a task, not a key result. "Decision turnaround time < 3 days" is a measurable outcome. Before finalizing OKRs, test each KR: "Could someone else observe and verify this was achieved?" If not, rewrite it.
59. **PMs need their own scorecard for CEO visibility** — Team KPIs (sprint velocity, bug rate) measure the team, not the PM. Add a dedicated PM Scorecard with metrics the CEO evaluates the PM on: decision turnaround time, scope change control, meeting compliance, stakeholder satisfaction pulse. Without this, the PM is invisible in their own dashboard.
60. **localStorage snapshots enable trend tracking without a backend** — Save a weekly snapshot object into a history array on every "Save" click. Use inline SVG sparklines (no Chart.js dependency) to show week-over-week progress. Cap at 13 weeks (~500KB). Add Export/Import JSON for backup. This gives trend visibility with zero infrastructure.

## OKR Dashboard v2 Lessons (Mar 25, 2026)

75. **Tab-based navigation beats sidebar for dashboards under 10 views** — v1 used 12 sidebar items; v2 consolidated to 7 tabs. Fewer navigation choices = faster cognitive load for executives. Only use sidebar when you have 10+ sections or need persistent visibility of all options.
76. **Bilingual i18n must use data attributes, not duplicate HTML** — v2 uses `data-i18n` keys with a JS translation map (~223 keys). Duplicating entire pages for EN/AR doubles maintenance and guarantees drift. Plan for `data-i18n` from the start so every text element is toggle-ready.
77. **Print CSS must show all tabs, not just the active one** — Dashboard print styles that hide inactive tabs produce blank pages. Use `display: block !important` on all `.tab-content`, add `page-break-before: always` between sections, and inject section titles via `content: attr(data-tab-title)` pseudo-elements. Always test Ctrl+P after adding print styles.
78. **Metric cards should be clickable drill-downs, not dead-end numbers** — Showing "To Do: 12" is useless without knowing who owns what. Every aggregate metric card should open a per-member or per-project breakdown modal on click (`showTeamDetailModal`). This was the key UX improvement in Team Pulse v2 — all 6 teams, 4 states each, 24 clickable drill-downs.
79. **Declare cached variables BEFORE loadData() runs** — JS `var` hoisting hoists the declaration but not the `= {}` initializer. If `var cache = {}` appears after `loadData()`, the initializer overwrites whatever `loadData()` restored from localStorage. Always declare cache variables before any load function that populates them.
80. **Never duplicate team members across teams** — Double-counting inflates metrics silently and makes drill-downs misleading. One person = one team. If they work across teams, pick their primary. OKR v2 had 4 members in both Mobile and Qatar — split them to eliminate overlap.
81. **WIQL returns work item IDs for free — don't discard them** — `data.workItems` from Azure DevOps contains `[{id, url}]`. The original `runWIQL` threw away IDs and returned only `.length`. `runWIQLWithIds` preserves them at zero extra API cost. Cache IDs in localStorage so drill-down links survive page reloads.
82. **Dashboards need a built-in user guide** — A PM won't use a dashboard they don't understand. Embed a step-by-step weekly workflow, a timeline with deadlines, and an auto-vs-manual field reference directly in the dashboard. Don't rely on external documentation — the guide must live inside the HTML.
83. **Always open HTML after edits — no exceptions** — Never leave the user to open files manually. Run `Start-Process` or equivalent as the final step after every HTML edit session. This was explicitly called out as a failure and saved as a feedback memory.

## Auto-Updater & UX Redesign Lessons (Mar 25, 2026)

84. **PowerShell em-dash crash** — Unicode characters (em-dash, curly quotes) in PowerShell string literals cause silent parse failures. The window closes instantly with no error. Always use ASCII alternatives (`--` not `—`, straight quotes not curly).
85. **Always provide .bat launcher for PS scripts** — "Right-click → Run with PowerShell" closes the window on any error. Create a `.bat` file with `powershell -ExecutionPolicy Bypass -NoExit -File script.ps1` so errors stay visible.
86. **Disable update buttons when no drift** — "Full Update" when already up-to-date is misleading and risky. Separate into 3 buttons: Check (always available), Pull+Update (only when drift), Audit Only (always available). Grey out unavailable actions.
87. **Automate recurring failures** — If a check is missed 3+ times across sessions, build a tool. The auto-updater script exists because manual drift fixes repeatedly missed Document Control tables, assumptions, and internal content. Human checklists fail; scripts don't forget.
88. **3 page types define nav UX** — Action pages (green, interactive buttons for decisions), Reference pages (blue, read-only for board review), Audit pages (purple, health history snapshots). Organize navigation by type, not by arbitrary categories. Directors need to know: "where do I click?" vs "where do I read?"
89. **Merge overlapping pages** — Two pages covering the same topic (KICKOFF_READINESS_GUIDE + KICKOFF_READINESS_VALIDATION) confuse the Director. One source of truth per topic. If you create a summary page, merge it into the original rather than adding another file.

## Cross-File Consistency & Audit Lessons (Mar 25, 2026)

90. **Deleting a file breaks all relative links to it** — When we deleted root-level HUB_PULL_HISTORY.html, 14 V11 sidebar nav links (`../HUB_PULL_HISTORY.html`) broke silently. Always grep for all references BEFORE deleting any file, and fix links in the same commit.
91. **Don't duplicate files across locations** — Having HUB_PULL_HISTORY at root AND inside V11 caused staleness (root was March 18, V11 was current). One copy, one location. If other files need it, link to it — don't copy it.
92. **Location-specific terms don't belong in hiring risks** — "Qatar market" implies on-site only. Hiring may be hybrid/remote. Use skill-based language instead: "Odoo 19 skill scarcity" not "Qatar market scarcity". Applies to all R-08 references across PM_RISK_REGISTER, ADR_APPROVAL_REQUEST, and any hiring-related text.
93. **Health dashboard verdict must reference cards by name** — A flat list of 7 pending items is unreadable. Group pending items under the card they belong to (e.g., "POC Readiness V.2 — ADR-1, Budget, Odoo hiring"). Board can then click the right card directly.
94. **Visual badges (ACTION NEEDED / CLEAR) save board reading time** — 7 cards with no visual priority = Board reads all 7. Amber badge + green badge = Board reads only the 3 that matter. Always add status badges when cards have mixed states.
95. **Track review loops explicitly** — The BA review between El-Afify and Mohamed had no visible status on the dashboard. Pass 2 was pending but nowhere documented. Add "Review Loop Status" tracking boxes to both the health dashboard card and the readiness page.
96. **Purpose tips in headers orient the reader** — Every page header should have a one-line description of what this page does. Board members open 15+ files — a tip below the title saves them from scrolling to understand context. Use `.summary` div pattern.
97. **Auto-updater covers KPIs but not consistency** — The PS1 server catches number drift (stories, SP) but misses text mismatches ("Qatar market" in 2 files), broken nav links, file count disagreements, and version tag staleness. Build a 6-check audit: KPIs + text + nav + file count + status sync + version tags.
98. **Sprint-plan-phase0 and phase0-poc-readiness are a pair, not duplicates** — Sprint plan = developer execution guide (Mohamed's, 2,196 lines). POC Readiness = PM status dashboard (ours, 227 lines). Different audiences, same data. Keep both but verify data consistency between them.
99. **Session handoff must include tracking file updates** — HUB_PULL_HISTORY, FOLDER_OBJECTIVE_MAP, and MEMORY.md must be updated at end of every session. Save a "FIRST TASKS Next Session" section in memory if context runs out.
203. **Supporting documents drift silently during version bumps** — Approval emails, cover letters, and non-artifact HTML files are not included in version-bump agent scope. KICKOFF_APPROVAL_EMAIL had V5 data (commit 2f321a3, March 9) while all 13 PM artifacts were at V13 (March 30). Stale numbers (18 vs 19 risks, 21 vs 28 actions, 200+ vs 937 stories) went unnoticed for 4 versions. Include ALL HTML files in audit-check scope.
204. **KPI smart cards must match source file buttons and KPI strips** — ADR_APPROVAL_REQUEST showed "1 Approved / 2 Pending" in KPI cards while all 3 ADR buttons were already "Approved." Manual review missed this; only the audit-check agent caught it. Smart card values are high-visibility — board sees them first — so mismatches create credibility risk.
205. **Internal summary text drifts independently from KPI cards in the same file** — PROJECT_STARTUP_CHECKLIST summary said "21 actions / 3 phases" while its own KPI cards correctly showed "28 / 4." PM_RISK_REGISTER meta said "18 Risks" while its KPI card showed "19." Always audit both the KPI strip AND the subtitle/meta/section headers — they are updated by different processes and diverge silently.
206. **Never expose internal PM governance file names on Hub V2 pages** — V13 Standalone file names (PM_RACI_MATRIX.html, HIRING_TEAM_PLAN.html) were added to phase0-poc-readiness.html on Hub V2. The team sees Hub V2 only; V13 references leak authority structure and create confusion about which files are canonical. Always use Hub V2 paths on Hub V2 pages.
207. **Hub V2 technical files created by engineering are NOT PM governance files** — sprint-plan-phase0 Key Contacts table (4 people) was treated as the project team structure. It's only a Phase 0 quick-reference. Hub V2 raci-matrix.html and hiring-team-plan.html are NOT approved by IT PM. Always confirm approval status before treating any file as authoritative for roles and responsibilities.
208. **Discrepancy reports need owner assignments + clickable source file links** — Review summary of sprint-plan-phase0 found 9 issues but initially had no owner or file references. Adding "Responsible: Mohamed Basha" + hyperlinked source files made it actionable. Without these, findings sit unresolved because nobody knows who should fix what or where.
209. **AI-generated placeholder names in Hub V2 files must be caught and flagged** — pbi-validation-phase0.html listed "Dr. Sarah Pemberton (BA)" as reviewer — a fabricated name. The actual reviewer is Mahmoud Afifiy (El-Afify). Placeholder names erode credibility if they reach board review and can cause confusion in formal audit trails.
210. **Separate internal evaluator content into local-only files before pushing** — Reference solutions (code answers) and interview questions (with expected answers) must NOT be in files pushed to shared repos. Candidates can see anything in the repo. Extract to V13-only files (e.g., ASSESSMENT_REFERENCE_SOLUTION_INTERNAL.html) and replace with locked placeholder in Hub V2 version.
211. **When moving files between folders, grep for ALL relative path references** — Moving team-assessment-phase0.html from Hub V2 to Team Assessment folder broke 8 CSS paths, 4 JS paths, and 3 cross-file links silently. Page rendered unstyled. Always grep for the filename across ALL files and fix paths in the same step as the move.

## HTML Asset Loading & Image Fallbacks (Apr 12, 2026)

328. **Always null-guard `onerror` on `<img>` fallback handlers — `onerror="this.src='fallback.jpg'"` is an infinite-loop bug** — Relief Center SLA v8 had `<img src="../Logo.jpg" onerror="this.src='../Logo -.jpeg'">`. When the primary failed, `onerror` reassigned `src`; if the fallback also failed (or the reassignment re-triggered the error), the handler kept firing because `onerror` was never cleared. The browser entered a load→fail→reassign loop, producing rapid flicker (logo appearing and disappearing multiple times per second). **Rule**: every `onerror` fallback MUST clear itself first — `onerror="this.onerror=null;this.src='fallback.jpg'"`. Apply to all `<img>`, `<script>`, `<link>` fallback patterns.
329. **Inline critical visual assets as base64 data URIs in client-facing HTML deliverables** — Relative paths like `../Logo.jpg` break when (a) the parent folder name contains spaces (`Relief Center` → URL-encoding inconsistency across browsers/file:// contexts), (b) the file is moved or emailed, or (c) the client opens it from a different location. Embedding the logo as `data:image/jpeg;base64,...` directly in the `<img src>` makes the HTML self-contained and portable. Cost: +44 KB on a single-file deliverable — negligible. Benefit: zero broken images regardless of where the file lives. Apply to: logos, signature marks, small icons in any standalone HTML deliverable sent to clients.
330. **Spaces in parent folder names silently break relative asset paths in file:// HTML** — Even when the file exists at the resolved path, some browser/OS combinations fail to load `../Logo.jpg` from a folder named `Relief Center` because of inconsistent space handling (`%20` vs raw space vs `+`). The failure is intermittent and hard to reproduce, which is the worst kind. **Two fixes**: (1) URL-encode spaces explicitly in `src` attributes (`../Logo%20-.jpeg`), or (2) inline the asset as base64 to eliminate the path entirely. Prefer #2 for client deliverables.
331. **When patching a versioned deliverable, also patch every co-existing sibling copy carrying the same bug — `glob` the folder before declaring done** — The Relief Center SLA logo flicker was fixed in `Relief_Center_SLA_v9.html`, but a parallel client-named copy `Relief Center & Pearl Pixels Services level Agreement (SLA).html` lived in the same folder with the identical buggy `<img>` tag. The user opened the client-named copy (not v9), the loop ran for hours in their browser tab, and 111,138 requests accumulated before they reported it again. **Rule**: after fixing a bug in a versioned file, immediately `grep` the entire deliverable folder for the buggy pattern (`onerror`, the broken path, the bad function call) and patch every match in the same batch. Versioning the canonical file does NOT propagate fixes to client-named or email-attachment copies. Applies to: SLA documents, dashboards, proposals, any folder where multiple naming conventions co-exist.
332. **Browser tabs with infinite-loop bugs keep running until the tab is killed — fixing the file on disk does not stop the loop** — After patching the buggy HTML on disk, the user's browser tab was still racking up requests because the page in memory was the pre-fix version. Reload alone may not be enough if the tab is hung; the fix must include "close the tab completely, then reopen". When reporting any infinite-loop fix to the user, the verification instructions MUST start with "close the existing tab" — never assume a refresh is sufficient.

For the first archival of a version chain, create the archive subfolder in that same turn and ask the user for the folder name once (then reuse it verbatim for all future bumps of that project — respect the user's spelling even if misspelled, e.g., "Old Vesrion"). For projects where an archive folder already exists, skip the naming question and just move.

## PM Governance Execution & External Handover (Apr 14, 2026)

380. **Content authorship vs mechanical metadata — governance override rule** — The "PM doesn't directly edit team-owned files" rule (lesson #370) exists to protect **content authorship** where owner expertise is required. A purely mechanical change — renaming a team member, fixing a typo, updating a date, replacing a stale email address — is **metadata maintenance**, not authorship, and the PM can execute it directly. **Why**: Apr 14, 2026 — CR-07 assigned an Amr Atya → Mohammed Chtioui rename across 10 team-owned files to 3 different owners (BA + SA + PO), which would have sat Pending for days waiting for 3 separate trivial PRs. 381. **Add new CR items to the open CR document — don't spawn a second one** — When a new cross-cutting issue surfaces mid-sprint and an open Change Request document already exists, add a **new row** to it rather than creating a separate CR file. 382. **Standalone-ify pattern for external handover copies** — When sending a repo-internal HTML document to an external team (UI/UX agency, client, auditor), create a standalone copy in a separate folder outside the repo and **strip all repo-specific dependencies** before sending: `<script src="nav.js">`, `<link rel="stylesheet" href="comments.css">`, `<script src="comments.js">`, any `<script id="embedded-comments">` JSON blob, and the `<nav class="nav-bar"></nav>` element. 386. **Email drafts save as `.html` not `.md` — Outlook copy-paste requires preserved formatting** — When drafting an email that will be sent via Outlook (or any Microsoft 365 mail client), save the draft as `.html` rather than `.md`. ## Azure DevOps Batch Work-Item Creation (Apr 15, 2026)

387. **`wit_add_child_work_items` fails silently on projects with required `System.AssignedTo` — fall back to per-item `wit_create_work_item` + single link batch** — The Azure DevOps MCP tool `wit_add_child_work_items` only accepts `title`, `description`, `areaPath`, `iterationPath` per child. It does NOT accept `System.AssignedTo`, `Microsoft.VSTS.Common.Priority`, `System.Tags`, or any arbitrary field. If the target project has a workflow rule requiring AssignedTo at creation time (common in Agile/Scrum templates with mandatory ownership), every single child in the batch fails with HTTP 400 `TF401320: Rule Error for field Assigned To. Error code: Required, HasValues, LimitedToValues, AllowsOldValue, InvalidEmpty` — and the error comes back as an array of 32 identical rejections, not a single top-level failure, which is easy to miss on a quick read of the response. **Why**: Apr 15, 2026 — attempted a 32-task batch creation under Relief Center PBI #4217 using `wit_add_child_work_items`. All 32 failed with the AssignedTo rule error because the Relief Center project requires AssignedTo at creation. Had to pivot to 32 individual `wit_create_work_item` calls (which accept arbitrary fields via the `fields` array), then one `wit_work_items_link` batch call to attach all 32 as children of 4217. Total recovery: ~40 seconds of parallel tool calls vs what would have been a clean single-batch if the tool supported it. **How to apply**: (1) Before any batch child creation, check the project's work-item type rules — grep past lessons or project memory for "AssignedTo required" flags; (2) If AssignedTo is required OR if you need to set Priority / Tags / custom fields at creation → skip `wit_add_child_work_items` entirely; (3) Use `wit_create_work_item` in parallel (send 10-15 tool calls in one message — confirmed no rate-limit issues on the TaqaTechno org), each with the full `fields` array including `System.AssignedTo`, `Microsoft.VSTS.Common.Priority`, `System.IterationPath`, `System.AreaPath`; (4) Collect all returned IDs, then make ONE `wit_work_items_link` call with `type: "child"` and `id: <parent>` + `linkToId: <new task>` for every new task — this attaches all children in a single API round-trip and is much cheaper than 32 sequential link calls; (5) Record the project's required-fields in the project memory file (e.g., `feedback_devops_required_fields.md`) so future sessions don't repeat the failed batch attempt. **Known projects with AssignedTo required**: Relief Center (confirmed Apr 15, 2026). Add others as encountered.

## Workspace Reorganization & Session Hooks (Apr 15, 2026)

## Browser API Call Throttling & Dashboard Stability (Apr 16, 2026)

391. **Never fire more than 6 concurrent Azure DevOps API calls from a browser dashboard — use a throttle pool** — When a browser-based HTML dashboard queries Azure DevOps via `fetch()`, never use `Promise.all()` with eagerly-created promises. Instead, wrap each call in a lazy task factory (`function() { return fetch(...) }`) and run them through a concurrency-limited pool (max 6, matching Chrome's per-domain connection limit). **Why**: Apr 16, 2026 — `Project Management Handling.html` fired ~290 simultaneous API requests on a single "Fetch from DevOps" click (11 projects x 7 query types + 7 teams x 6 queries + 20 members x 6 queries + batch fetches). This exhausted browser memory (2-4 GB spike), froze the network stack, and crashed the entire laptop — requiring 2 forced restarts. Root pattern: 5 nested `Promise.all()` arrays all created eagerly, meaning all HTTP connections were initiated at construction time with no backpressure. **How to apply**: (1) define a `throttledRun(taskFactories, limit)` function that takes an array of zero-argument functions returning promises, starts `limit` concurrently, and launches the next task only when a slot frees up; (2) convert every `someArray.push(fetch(...))` to `someArray.push(function() { return fetch(...) })`; (3) replace `Promise.all(someArray)` with `throttledRun(someArray, 6)`; (4) add an abort flag checked between task launches so users can cancel mid-fetch; (5) show a progress counter ("X/N queries") so users know the fetch is alive. The throttle adds ~30-40s to total fetch time (sequential vs parallel) but prevents system instability. For dashboards with 100+ queries, consider a server-side cache (e.g., `DevOps_Cache_Server.ps1` on localhost:5051) that fetches once and serves cached JSON to all HTML consumers.

392. **Always add an Abort button to long-running browser fetch operations** — Any browser fetch operation that fires more than ~10 API calls should include a visible "Abort" button that sets a cancellation flag, stops new requests from launching, and cleanly restores the UI (re-enable the fetch button, show "Aborted" status). Without an abort mechanism, the user's only escape from a stuck or slow fetch is to kill the browser process — which on a resource-constrained machine can require a full system restart. **Why**: Apr 16, 2026 — `Project Management Handling.html` had no abort mechanism. Once "Fetch from DevOps" was clicked, ~290 requests were queued with no way to cancel. The user's laptop hung and required 2 forced restarts. After adding an `_abortFetch` flag checked by `throttledRun()` between task launches, plus a red "Abort" button injected next to the fetch button, the user can cancel at any time and the UI recovers cleanly. **How to apply**: (1) declare a module-level `_abortFetch = false` flag; (2) set it to `false` at the start of each fetch; (3) check it inside the throttle pool's `next()` function — if true, reject the promise chain; (4) create/show an "Abort" button (red, next to the fetch trigger) when fetch starts; (5) hide the abort button and re-enable the fetch button in both the success and error handlers; (6) in the error handler, check if the error was an abort and show a distinct "Fetch aborted by user" message instead of a generic error.

## PM-Dashboard Platform Build (Apr 16, 2026)

393. **When multiple PowerShell HTTP listeners serve the same user, consolidate into one lightweight server** — On Apr 16, 2026 the PM workspace had 4 separate PowerShell `HttpListener` servers on ports 5050-5053 (Outlook search, DevOps cache, drive scanner, reserved), each requiring manual startup via right-click `.bat` files. All shared the same `Send-Json` helper function copy-pasted across files. Consolidating into a single Bun + Hono server on port 3000 replaced all 4 with one `start.bat`, eliminated code duplication, and enabled shared SQLite state across all endpoints. **Why:** Each additional listener is a manual startup step the user forgets before meetings, a port conflict risk, and a maintenance burden when the shared pattern (CORS headers, JSON serialization) changes. **How to apply:** When you find 2+ local HTTP servers serving the same user/browser, unify them behind one server with route prefixes (`/api/cache`, `/api/search`, `/api/scan`). Use a framework like Hono that adds ~15KB overhead. Keep the old scripts as fallback until the new server is proven.

394. **Bun + Hono + bun:sqlite is the optimal stack for single-PM local dashboards on Windows** — After evaluating Grafana (200MB, no RTL), Metabase (requires Java), Evidence.dev (requires rewriting HTML to Markdown), n8n (Docker, overkill), and Astro/Next.js (build step complexity), the best fit for a Windows PM who works in Claude Code is Bun (10ms startup, built-in SQLite, TypeScript) + Hono (15KB server) + bun:sqlite (zero-dependency DB). **Why:** The PM is not a full-time developer — the stack must be maintainable by someone who codes through Claude Code, not by hand. Bun's built-in SQLite eliminates native module compilation issues on Windows. Hono's API is simple enough that Claude can scaffold and maintain the entire server. **How to apply:** For any local-first dashboard project with <100K data rows, Azure DevOps as data source, and a single PM user — default to this stack. Skip React/Vue/Angular entirely; vanilla JS + ECharts serves the same purpose without build tooling.

395. **Move PAT tokens from browser localStorage to server-side .env files immediately** — On Apr 16, 2026 the PM workspace had `localStorage.getItem('devops-pat')` returning the Azure DevOps PAT in cleartext across 3+ HTML dashboards. Any browser extension or same-origin script could read it. Moving the PAT to a `.env` file read by the Bun server eliminated this exposure — the browser never sees the token, only calls `localhost:3000/api/*` which adds auth headers server-side. **Why:** localStorage has no access control, no encryption, and persists across browser sessions. A leaked PAT grants full Azure DevOps access. **How to apply:** (1) Create `.env` with `AZURE_DEVOPS_PAT=...`; (2) Add `.env` to `.gitignore`; (3) Server reads PAT via `process.env` and adds `Authorization: Basic` header on outbound requests; (4) Browser fetches from `localhost/api/*` with no auth header; (5) Remove PAT modal from HTML dashboards (keep as fallback for direct DevOps access).

396. **localStorage is not persistence — SQLite is the minimum viable data store for dashboards** — On Apr 16, 2026 all PM dashboard state (OKR scores, health statuses, Data Gate snapshots, SPI/CPI calculations, portfolio narratives) lived exclusively in browser localStorage under 15+ keys. Clearing browser data, switching browsers, or a browser crash wiped everything with no recovery path. The only backup was a manual "Export JSON" button. Replacing localStorage with SQLite (`pm-dashboard.db`) gave file-system persistence, historical snapshots, cross-dashboard shared state via API, and a single backup target. **Why:** localStorage is per-origin, per-browser, volatile, and has a 5-10MB limit. SQLite is a single file that survives browser changes, can be backed up with a file copy, and supports SQL queries for trend analysis. **How to apply:** For any dashboard that stores user-entered data beyond session scope, use SQLite (via `bun:sqlite` or `better-sqlite3`). Expose state via `GET/PUT /api/state/:key` endpoints. Keep localStorage as a read cache for offline scenarios, but treat SQLite as the source of truth.

397. **When migrating HTML dashboards to a new API, keep the old fetch path as fallback** — On Apr 16, 2026 OKR v7 was migrated to fetch from `localhost:3000/api/cache` instead of direct Azure DevOps calls. Instead of replacing the existing "Fetch from DevOps" button, a second "Fetch via Cache" button was added alongside it. This meant if the Bun server was down, the user could still fetch directly from Azure DevOps. **Why:** The new server is a single point of failure. If it crashes, all dashboards that depend on it become read-only. Keeping the direct-fetch fallback ensures the PM can always get fresh data for a meeting, even without the server. **How to apply:** When adding a cache/proxy layer in front of a direct API, add a new button/path rather than replacing the old one. 398. **Auto-start local servers via Windows Startup folder shortcut, not VBS or Task Scheduler** — On Apr 16, 2026 three approaches were tried for auto-starting the PM-Dashboard server on Windows login: (1) `schtasks` — failed with "Access denied" (needs admin); (2) VBS script in Startup folder — failed with "Not enough memory resources"; (3) `.lnk` shortcut pointing to a `.bat` file with `WindowStyle=7` (minimized) — worked reliably. **Why:** Task Scheduler requires admin elevation. VBS `WScript.Shell.Run` is fragile on Windows 11 with memory pressure. 399. **Index CSV email data into SQLite at server startup for instant search** — On Apr 16, 2026 the Outlook email search on port 5050 scanned 12 CSV files (3,342 rows) on every search request using PowerShell `Import-Csv` + in-memory string matching. Moving the index to SQLite at server startup (136ms one-time cost) made every subsequent search instant (<10ms) and eliminated the need for the PowerShell server. **Why:** CSV scanning is O(n) per query and requires the PowerShell server to be running. SQLite indexing is O(n) once at startup, then O(log n) per query, and the data persists across restarts. **How to apply:** At server startup, read all CSV files, parse with a simple CSV parser, and bulk-insert into an `email_index` table using a transaction. 400. **Historical snapshots in SQLite enable trend sparklines with zero additional infrastructure** — On Apr 16, 2026 every `POST /api/refresh` stored SPI/CPI and Data Gate values as timestamped rows in `spi_cpi_snapshots` and `data_gate_snapshots` tables. After just 2 refreshes, the `GET /api/trends/spi-cpi?days=30` endpoint returned enough data to render sparkline mini-charts (ECharts, 80x30px) in the SPI/CPI table. Over days/weeks, these accumulate into meaningful trend lines with no additional tooling. **Why:** Most trend visualization requires a time-series database (InfluxDB, Prometheus). But for a single PM with <1000 data points per month, SQLite `INSERT + SELECT WHERE snapshot_at >= ?` is sufficient. The key insight: store snapshots on every refresh, not just the latest value. **How to apply:** For any metric that changes over time (SPI, CPI, gap%, task counts), always INSERT a new row with a timestamp rather than UPDATE the latest row. ## Ragtools Operational Hardening — Watchdog Registration & Boot Readiness (Apr 17, 2026)

430. **Writing a watchdog script ≠ registering it — verify with Get-ScheduledTask, not by reading the script header** — On 2026-04-17, investigation of "RAG not working after reboot" revealed `C:\Users\DELL\AppData\Local\ragtools\Watch-RAGService.ps1` (written Apr 16) had a comment at the top reading "Registered via Task Scheduler to run at login", but `schtasks /query` showed no such task existed. `watchdog.log` contained exactly one line from Apr 16 16:10:30 when it was launched manually, then nothing — it died with that shell and was never resurrected. For 24+ hours the user believed "the watchdog protects me" while it was doing nothing. Fix was a single `Register-ScheduledTask` call. **Why:** Shell scripts, hooks, and watchdog scripts are only active when *something else* — Task Scheduler, systemd, a Run registry key, a Startup shortcut, a service wrapper — is configured to run them. Writing the file is maybe 20% of the work; wiring it into the runtime is the other 80%. Comments inside the script describing its intended deployment are intent, not proof. **How to apply:** For every "run at login / on a timer / at service start" script, the commit that adds the script must also add the scheduler entry that actually runs it (and a verification command that confirms it). Acceptance test on Windows: `Get-ScheduledTask -TaskName '<name>'` returns `Ready` or `Running`. On Linux: `systemctl is-enabled <name>` returns `enabled`. On macOS: `launchctl list | grep <label>`. If the verification command returns nothing, the script is not deployed — regardless of what the header says. Pair every new watchdog/autorun script with its register-and-verify commands in the same checklist.

431. **"Service started" ≠ "service ready" — autostart scripts must poll the readiness endpoint before exiting** — On 2026-04-17, `RAGTools.vbs` in the Startup folder called `rag.exe service start` and exited immediately. The CLI returns in milliseconds, but the HTTP service on port 21420 took 3–4 minutes to become responsive (encoder load + 20,968 Qdrant points — past the 20k local-mode recommended limit documented in `config.toml`). Every login had an invisible 1–2 minute dead window where MCP queries failed silently before RAG became available. Fix was to add a polling loop after `service start`: `For i = 1 To 12 / WScript.Sleep 5000 / MSXML2.XMLHTTP GET /health / Exit For on 200`, capped at 60s. **Why:** Service start commands typically return as soon as the child process is launched, not when the service is actually listening. Any downstream caller that assumes "start returned → service ready" hits a race. For heavyweight services (ML model load, large vector-store init, multi-GB cache warmup), readiness can lag process launch by minutes. The only reliable signal that a service is ready is a 200 response from its own health endpoint. **How to apply:** In every autostart / service-launch script, after issuing `start`, loop: sleep N seconds → probe `/health` (or the service's readiness endpoint) → exit success on 200 → timeout with a loud error after M minutes. Never exit success based on "start command returned". Log the wall-clock gap between start-issued and health-ready — if it grows, your cache/model/index has crossed a size threshold and the load time is now a user-visible delay worth mitigating (docker, remote backend, on-demand init, etc.). For ragtools specifically: the config.toml warning at 20k Qdrant points matched the observed 3–4 min load.

432. **"Not recognized" errors mean wrong shell, not wrong command — and full-buffer pastes corrupt the next shell** — On 2026-04-17, the user ran `Stop-Process -Name rag -Force` in `cmd.exe` (Command Prompt) and got `'Stop-Process' is not recognized as an internal or external command`. `Stop-Process` is a PowerShell cmdlet; CMD has no access to it. The user's next attempt was to paste the *entire* CMD window contents — banner `Microsoft Windows [Version 10.0...]`, copyright line, previous prompt, error text, everything — into PowerShell. PowerShell tried to parse the banner as code and emitted `Unexpected token 'Microsoft' in expression or statement`. Both errors looked like the command was wrong; neither was. The first was a wrong-shell problem; the second was a paste-hygiene problem. **Why:** Command names work differently across shells — PowerShell cmdlets aren't in CMD, bash utilities aren't in PowerShell, Unix-style flags aren't in Windows shells. When a user copies from one terminal and pastes into another, they often grab the full buffer including prompt text, which the target shell tries to execute. The resulting parser errors look like syntax problems but are really shell-identity and paste-hygiene problems. Blaming the command wastes time. **How to apply:** (1) When a shell says "not recognized" or emits `Unexpected token '<English-word>'`, first diagnose **which shell** the command is meant for: PowerShell cmdlets (`Stop-Process`, `Get-ChildItem`, `Invoke-WebRequest`) need a `PS C:\>` prompt; CMD commands (`dir`, `taskkill`, `where`) need `C:\>`; bash commands (`ls`, `grep`, `curl`) need a Unix shell. ## Cadence Migration, i18n Edge Cases & CWD-Lock Workarounds (Apr 20, 2026 — continued)

459. **Audit BOTH literal-Unicode AND `\uXXXX` escape forms when bulk-replacing non-ASCII text in a single file** — On 2026-04-20, migrating the PM Handling dashboard from bi-weekly to weekly cadence required replacing Arabic phrases like "نصف الشهرية" → "الأسبوعية" and "3 مايو" → "26 أبريل". A single JS file mixed both forms: `PORTFOLIO_NARRATIVES` entries used literal Unicode chars (the user typed them directly), while the T i18n dictionary used `\u0645\u0627\u064a\u0648`-style escape sequences (copy-pasted from older code generators). My first Node pass with literal-Arabic replacements caught ~70% of occurrences; the remaining ~30% needed a second pass targeting the escape form explicitly. Took 3 passes + 1 manual edit to reach 0 residual. **Why:** File tools see bytes, not characters. The literal-char form `\u064a\u0648` as Unicode scalar U+064A U+0648 (a 4-byte UTF-8 sequence) is COMPLETELY different from the ASCII escape sequence `\` + `u` + `0` + `6` + `4` + `A` + `u` + `0` + `6` + `4` + `8` (11 ASCII bytes). A string-replace targeting one form silently skips the other. Long-lived codebases accumulate both forms because different contributors/tools wrote them differently. **How to apply:** (1) Pre-flight grep: `grep -c "LITERAL_ARABIC_WORD" FILE` AND `grep -cE "\\\\u[0-9a-f]{4}.*\\\\u[0-9a-f]{4}" FILE` — if both non-zero, the file has mixed encoding. (2) For bulk rename, write TWO replacement tables: one for literal chars, one for `\u`-escape form. (3) Run both passes. (4) Final grep for residuals using the escape form of each target word (e.g. `\u0645\u0627\u064a\u0648` for مايو). (5) If you're generating new content, prefer ONE form (literal UTF-8 is more human-readable) and don't perpetuate the mix.

460. **Arabic (and other Semitic languages) gender/state inflections demand their own replacement rules — don't assume one form covers all** — On 2026-04-20, my cadence migration replacement table handled the feminine definite form `نصف الشهرية` but missed two other forms that appeared in the same file: the masculine definite `نصف الشهري` (no ة on the adjective — because "اجتماع" is masculine and agreement requires a masculine adjective) and the indefinite feminine `نصف شهرية` (without the definite article ال). A roadmap-footer sentence reading "يتم مراجعة التقدم في اجتماع 3 مايو نصف الشهري" (progress is reviewed in the May 3 bi-weekly meeting) survived both automated passes because the adjective was masculine (`الشهري` without ة) — my pattern was `نصف الشهرية` with ة. Required a targeted manual edit. **Why:** Semitic-language adjectives inflect for gender (masc/fem), state (definite/indefinite), number (singular/dual/plural), and case (nominative/accusative/genitive — in classical Arabic). A single English word like "bi-weekly" can correspond to 4+ Arabic surface forms depending on what noun it modifies: `نصف شهري / نصف شهرية / نصف الشهري / نصف الشهرية / نصف شهريان / نصف الشهريان / ...` Naive string-replace hits one form and leaves the rest. The same class of issue exists for Hebrew, French, German, Spanish (adjective agreement), Russian (case endings), etc. **How to apply:** (1) Before writing a bulk-rename script, grep the file for the ROOT noun (e.g., `شهري`, `أسبوعي`) not the full phrase — list every inflected form you find. (2) Build the replacement table with one pair per inflection: `(نصف الشهرية, الأسبوعية)`, `(نصف الشهري, الأسبوعي)`, `(نصف شهرية, أسبوعية)`, `(نصف شهري, أسبوعي)`. (3) After the automated pass, grep for the root noun one more time — any survivors are forms you missed. (4) For content PMs write themselves, keep an internal glossary of the forms each acronym takes (see lesson #457 on the bilingual PM vocabulary reference).

461. **Phrase-level replacement tables must include disambiguating context — a bare date or number match across hundreds of files corrupts calendar-week markers** — On 2026-04-20, the bi-weekly→weekly migration needed to change the next-review-target date "May 3" → "Apr 26". The v3 HTML contains "May 3" in 13 places, but only 11 were deadline references to update. The remaining 2 were calendar-week timeline rows: `<tr>W18 Apr 27 – May 3 Weekly update</tr>` and `<tr>W21–W22 May 18 – May 31 Weekly updates</tr>`. A bare `'May 3' → 'Apr 26'` find-and-replace would corrupt the W18 week into a nonsense `Apr 27 – Apr 26` span. Fix: every entry in the replacement table included disambiguating context — `'by May 3 review'`, `'by May 3.'`, `'Target ≥ 60% by May 3'`, `'Sunday May 3'`, `'May 3 bi-weekly review'` — never bare `'May 3'`. **Why:** A date string like "May 3" is semantically overloaded: it can be a deadline, a calendar week endpoint, a historical event marker, a meeting name, a sprint close, a milestone. Each semantic needs a different fate during bulk migration. The context before/after the date disambiguates. Naive global find-and-replace is the second-easiest way to silently corrupt data (after accidentally deleting something). **How to apply:** (1) Before writing any bulk-rename rule for a date, number, acronym, or name, grep the file for ALL occurrences of the bare string and categorize them by surrounding context. (3) After the pass, grep for bare occurrences of the original — residual hits should all be false positives (leave-alone cases) or clear misses worth a manual review. (4) For calendar-week vs deadline disambiguation specifically: week markers follow the pattern `W## <date> – <date>` or `Week N (<date>)`; deadlines follow `by <date>`, `<date> review`, `<date> sprint close`. 462. **When renaming an i18n key, update THREE locations in one atomic pass: the HTML `data-i18n` attribute, the dictionary lookup key, and the translation value — drift between any two breaks the render** — On 2026-04-20, changing the next-review date broke multiple i18n lookups because I updated the visible English text (between `<span data-i18n="...">...</span>` tags) but forgot to update the `data-i18n` attribute VALUE (the lookup key) OR the dictionary entry KEY string. Result when toggling AR: the browser looked up `T['Estimation Accuracy from SBI tab ≥ 60% by Apr 26 review']`, found no match (the dictionary still had `T['Estimation Accuracy from SBI tab ≥ 60% by May 3 review']`), and fell back to showing the EN text verbatim — looking "not translated" instead of translated. Also saw the opposite symptom: dictionary KEY was updated but the AR VALUE still contained the old date — AR mode then showed an Arabic string with the WRONG date. Took 3 passes to align all three. **Why:** The i18n contract is a three-way identity: `<span data-i18n="KEY">EN_TEXT</span>` PLUS `T[KEY] = AR_TEXT`. If any one of (EN_TEXT, KEY, AR_TEXT) changes without the other two, the contract breaks. Most UI i18n frameworks have a "key-first" pattern where translations are looked up by key, so dictionary key and HTML attribute MUST match byte-for-byte; the EN visible text is somewhat decoupled but visually should match the key for consistency. **How to apply:** (1) Treat the three locations as one atomic unit in any find-and-replace script. (2) After the pass, grep for the OLD form across all three contexts — if any survive, the rename is incomplete. (3) Verify via Playwright: toggle to AR and confirm the AR text appears (not the EN fallback). If the EN text shows in AR mode, the key/attribute/dictionary-key don't match. (4) Also update the AR VALUE in the dictionary if it contained the old date/term — the AR text has its own domain-specific strings that need updating in parallel.

464. **Windows refuses to rename or move a folder if any process's current-working-directory (CWD) is set to that folder OR any subfolder within it — work around by moving CONTENTS rather than the folder itself** — On 2026-04-20, attempting to archive `D:\Project Management\Project management Quarter Plan\Projects Handling Review Meeting 6-4-2026\` with `mv` returned `Device or resource busy` even after killing all `node.exe` HTTP server processes holding the folder open. Root cause: the current Claude Code session had this folder set as its CWD (from the original VSCode workspace-open), and Windows NTFS holds a handle on the CWD of every running process. Even an in-place rename was blocked. Workaround that worked: create the archive target, then MOVE each TOP-LEVEL ITEM out of the folder one-by-one, leaving only the tool-state subfolders (`.claude/`, `.playwright-mcp/`) that the session was actively using. Result: 35 items relocated, folder shell remains with 2 hidden tool-state dirs, clean separation between "archived content" and "active session state". **Why:** Windows NTFS differs from POSIX filesystems in how it handles folder locks. On Linux/macOS, an inode's path is rename-safe even when a process has CWD there — the process just continues holding the OLD path via its inode reference. On Windows, NTFS locks folders that are any process's CWD, and the rename fails. This affects `mv`, `ren`, `robocopy /MOVE`, `PowerShell Rename-Item`, and even `rmdir`. Subprocess trees inherit the parent's CWD, so spawning `bash` or `powershell` from a shell rooted in the folder compounds the issue — every child process also has a lock. **How to apply:** (1) Before attempting to move/rename a folder that's currently open in any editor/IDE/terminal/debugger, identify which processes have it as their CWD. `handle.exe -p <pid>` (Sysinternals) or `PowerShell Get-Process | Where-Object { $_.Path -like "*folder*" }`. (2) If you CAN close those processes (plain terminals, build watchers), close them and retry. (3) If you CAN'T (your own editor session's state folder), move CONTENTS instead of the folder. Use a loop: for each top-level item, `mv item TARGET/`. Skip hidden tool-state subfolders like `.git`, `.claude`, `.playwright-mcp`, `.vscode`, `node_modules` only if they belong to the active session. (4) After the content move, the empty shell can be renamed/deleted from a DIFFERENT session (one whose CWD is elsewhere). (5) For automation scripts that MUST rename folders, set `cd /` or `cd ~` at the start so the script's own CWD isn't inside the target. (6) For VSCode + Claude Code specifically: the session's workspace root is the CWD — move content only, delete the shell from a fresh session opened elsewhere.

## Heuristic Text Parsing & Data Pipeline Hygiene (Apr 21, 2026)

468. **Factual placeholder beats empty or invented when the source data is literally unrecoverable** — When a source artifact exists but cannot be read (e.g., image-only PDFs that fail text extraction with zero characters recovered, locked Outlook .msg files, corrupted attachments), the row/entry must still carry a label. The honest choice is a descriptive placeholder like `Unreadable CV (1776112376090.pdf)` — never a blank field, never a fabricated guess. **Why:** Two user-stated rules often collide — "don't leave any row without a name" AND "don't assume any name". A factual placeholder satisfies both: it's a real, verifiable string that tells the reader exactly what state the data is in. Invented values (blindly guessing a name from a filename stem) cross into fabrication; blank values violate the completeness guarantee and usually cascade into downstream joins and filters silently dropping rows. **How to apply:** (1) Detect unrecoverable cases at the extraction layer, not the rendering layer — mark them with a distinct confidence tag (e.g. `name_conf = "unparsed"`) so downstream consumers can treat them specially. (2) The placeholder string must include the source identifier (filename, URL, message ID) so the human reviewer can trace back. (3) Surface the unparsed count in the run summary — hiding them behind the placeholder is as bad as hiding them behind a blank. (4) Applies beyond CVs: any bulk ingest (emails, receipts, scanned invoices, OCR pipelines, satellite image metadata) with partial failure modes. Surfaced by the D:\HR candidate-name audit on Apr 21, 2026 — 3 of 51 fixed candidates had image-only PDFs with zero extracted text.

469. **Heuristic text extractors fail in three predictable classes — blocklist all three with complementary data structures** — On 2026-04-21, the candidate-name parser in `D:\HR\_pipeline\step2_parse_and_score.py` misread 51 names as (a) section headers like `Work History` / `Core Skills`, (b) education/credential phrases like `Bachelor of Computer Science` / `Future University in Egypt`, and (c) 2-word capitalized phrases that coincidentally look like names such as `Nasr City` / `Saudi Arabia`. The original heuristic had only ~25 section-header entries and missed all three classes. **Why:** Each class needs a different data structure. Exact-match sets catch single phrases ("work history"), substring-match sets catch variants ("bachelor of", "bachelor degree", "bachelor's degree" all reduce to a single "bachelor of" prefix), and per-word disqualifier sets catch any line containing a poisoned token. One list catches one class — trying to use a single exact-match set for all three was the root cause. **How to apply:** (1) Build three blocklist structures up front in any heuristic extractor: `SECTION_HEADERS` (exact match), `NON_NAME_SUBSTRINGS` / `NON_MATCH_SUBSTRINGS` (substring match for multi-word variants), and `DISQUALIFIERS` (per-word match). (2) Seed them from a real 10-20 case audit of actual production data — do not rely on imagination. (3) When a new false positive is reported, classify it first: is this an exact phrase, a multi-word variant, or a poisoned token? Add to the matching structure that fits — not to whichever list is most convenient. (4) Applies beyond names — the same pattern fits email subject-line classifiers, product-title parsers, and any regex-heuristic layer upstream of an ML model.

470. **Scan order matters in multi-pass parsers — specific/tight rules must run BEFORE broad/noisy rules** — In the extraction function `extract_name()`, running single-line separator-splitting on lines 1-50 BEFORE the "combine two consecutive short lines" pass caused regressions: a CV with "Ola" on line 1 and "Mohammed" on line 2 (each a single word needing consecutive-line combine) was instead parsed as "Nasr City" from line 9's address `Nasr City, Cairo, Egypt` — the broad rule matched first and short-circuited the specific one. Reordering — specific rule first, broad rule second — fixed this. **Why:** In heuristic parsers, the first passing rule wins. If a noisy/broad rule runs early, it will occasionally claim a false positive before the correct tight rule has a chance to fire. The intuitive ordering ("scan top-down, try every rule on every line") is wrong for rule pipelines — you want confidence-ordered, not location-ordered. **How to apply:** (1) List all heuristic passes with their precision/recall tradeoff. Tight rules (specific patterns, narrow scope) go first. Broad rules (fuzzy separators, wider scope) go last. (2) For each rule, enumerate what it *will* match that it shouldn't. If that list is non-empty and the next-tighter rule would catch those cases, reorder. (3) Run a diff harness after every rule-order change — the surface area for unintended matches is wide. (4) Applies to any ordered rule engine: spam filters, fraud-detection thresholds, CSS selector specificity, redirect rule lists, firewall rules.

471. **Scope fuzzy rules to the narrowest window where they pay off** — Separator-splitting on double-space `  `, `|`, `·`, `•`, `, ` is valuable on the CV header line ("Ahmed Abaza  Junior Quality Manager" → splits to `["Ahmed Abaza", "Junior Quality Manager"]`, first chunk is the name) but toxic on line 9 of the same CV ("Nasr City, Cairo, Egypt" → splits to `["Nasr City", "Cairo", "Egypt"]`, first chunk is a location). Same rule, different damage by scope. Limiting the separator-split pass to the first 5 lines (the header zone) eliminated all location false positives without losing the header wins. **Why:** Fuzzy rules carry both signal and noise; the signal-to-noise ratio depends on WHERE in the document they fire. Document structure gives you implicit scope information — headers are name-dense, body is fact-dense, footers are reference-dense. Applying a fuzzy rule uniformly across all regions ignores this and collects noise that cancels the signal. **How to apply:** (1) For every fuzzy/noisy rule, ask: "what is the narrowest span of the document where this rule pays off?" Default to lines 1-5 for CV headers, the first paragraph for news articles, the subject line for emails, the first screen for web pages. (2) Log where the rule fires during development — if >30% of fires are outside the intended zone, narrow the scope. (3) Never apply separator-splitting, letter-spaced recovery, or email-inference fallbacks globally — always scope them to the region they were designed for. (4) Applies to OCR post-processing, address extraction, author/byline detection, table-header detection, CSV-guessing heuristics.

472. **Arab/Egyptian compound names need hyphen-aware parsing — reject only lowercase-hyphenated phrases, never "any hyphen in a multi-word line"** — Names like `El-Absawy`, `Abd-Elhamid`, `Al-Zanaty`, `Abu-Bakr`, `Al-Hussein` must pass the name-plausibility check. The original rule "if line contains `-` AND has multiple words, reject" silently dropped every candidate with a hyphenated surname segment — forcing them into the email-fallback path and often producing incomplete or wrong names. The correct rule: "each hyphen-separated segment of a word must start with a capital letter". This accepts `El-Absawy` (both caps) and `Abu-Bakr` (both caps), but still rejects English compound phrases like `web-development`, `end-to-end`, `results-driven`, `object-oriented` (all lowercase). **Why:** In the MENA hiring pool, a material fraction of surnames contain an Arabic definite-article hyphen (`El-`, `Al-`, `Abd-`, `Abu-`, `Ibn-`). A blanket hyphen-reject rule discriminates against them by silently dropping their CVs into a fallback path with worse accuracy. The English lowercase-hyphen-phrase pattern is structurally different (lowercase-first, kebab-case) and can be distinguished with a per-segment capital check. **How to apply:** (1) In any name-validation routine, replace the "reject multi-word lines with hyphens" rule with a per-word check: for each word containing `-`, split on `-` and verify each segment's first character is uppercase. (2) Add explicit test cases for Arab compound names alongside the English-kebab rejection cases. (3) Applies to any global HR system, authorship metadata, directory listings, or ID verification that needs to handle multi-cultural name formats (also relevant for Spanish `de la`, Dutch `van der`, etc. — add those as supplementary rules).

473. **Regression-test the WHOLE corpus, not just the known-bad cases you set out to fix** — The 51 targeted name fixes in step2 silently introduced 4 regressions on previously-correct rows (Ola Mohammed, Amir El-Absawy, Software Quality Control, Saudi Arabia) because my before/after test harness only compared outcomes against the 51 known-bad fix targets. The 342 other candidates — whose names had already been parsing correctly — were outside the test loop, so a heuristic change that now mis-parses them went undetected until I scanned the final HTML output. Adding a "scan all output for suspicious patterns NOT in your fix list" gate caught the regressions and prompted the hyphen-rule and separator-scope fixes. **Why:** Targeted heuristic changes rarely stay targeted — any adjustment to a shared parser, filter, or scorer has blast radius across every row that touches the changed code path. Testing only the rows you *meant* to fix confirms you addressed the intended bug but gives zero signal on the rows you didn't mean to break. **How to apply:** (1) Before any heuristic change, snapshot the full current output (all rows, not just the bad ones). (2) After the change, diff the full new output against the snapshot. (3) Any row whose value changed AND that wasn't on your intended-fix list is a regression candidate — triage each one explicitly. (4) For production pipelines, automate this as a gate: fail the build if the diff touches rows outside the allowed changeset. (5) Applies to any rule/scorer/filter change — spam classifiers, fraud thresholds, tax-category mappers, pricing rules, permission matrices.

474. **Empty return beats wrong-but-plausible, when a fallback chain exists** — When a heuristic extractor returns `""` (empty), the next stage in the fallback chain (email-local-part inference, filename stem, structured-metadata lookup) can recover. When it returns "Nasr City" (a wrong-but-plausible 2-word phrase), the chain short-circuits because `""` is falsy but `"Nasr City"` is truthy — so the wrong answer ships. Tuning the primary heuristic toward restraint ("return empty if I'm not confident") is strictly better than tuning it toward coverage ("return something") for any pipeline that has downstream fallbacks. **Why:** Short-circuit chains (`result or fallback1 or fallback2 or default`) are common in Python, JavaScript, Ruby — they stop at the first truthy value. A falsy-but-honest empty is transparent to the chain; a truthy-but-wrong value is opaque and blocks recovery. The asymmetry is: a false negative (rejecting a real name) costs one fallback step; a false positive (returning a fake name) costs the entire correctness of that row. **How to apply:** (1) In any extractor feeding a fallback chain, bias toward `return None` / `return ""` over "best guess" when confidence is below a threshold. (2) Use an explicit confidence tag (`name_conf = "ok" | "email-derived" | "filename-fallback" | "unparsed"`) so downstream rendering can flag the provenance. (3) Log empty-return cases — they are not errors, they are correct honest outputs. (4) Do NOT apply this pattern when there is no fallback chain — a single-shot extractor must return its best guess or the row is lost. (5) Applies to OCR confidence thresholds, ML-classifier calibration (abstain-option), entity-resolution joins, and any parser/normalizer with a "don't know" option.

475. **Always keep a post-processor override layer for the long tail the heuristic can't reach** — Even a well-tuned heuristic name parser won't hit 100% on messy real-world CVs — graphic layouts that bury the name past contact/skills/certificates blocks, all-image PDFs, letter-spaced headers spanning 3 lines, multilingual headers with mixed scripts. After the Apr 21 step2 audit, the heuristic direct-hit rate for the 51 known-bad candidates went from 0/51 to 29/51 (57%) — a big win, but still leaves 22 cases needing help. The persistent solution is a small manual-override file (`D:\HR\_pipeline\fix_names.py`) mapping `filename → (corrected_name, name_conf)`, applied as the final pipeline step. It's idempotent, survives step2 re-runs, and lets the PM append new overrides in one line without touching the heuristic. **Why:** Heuristics have fundamental limits on messy natural-language data. Chasing 100% parser accuracy via more blocklists produces a parser that is overly conservative, breaks on legitimate new inputs, and still misses the truly hard cases. An override layer cleanly separates "rules that generalize" (in the parser) from "exceptions that don't" (in the override file), keeping each code path simple. **How to apply:** (1) Every extract/parse/transform pipeline with human-review output should have an override step as its last stage. (2) The override format should be plain data (dict, JSON, YAML, CSV) that a non-coder can edit — `{filename: name}`, `{id: status}`, `{key: value}` — not a script. (3) The override layer applies unconditionally; it does NOT check whether the upstream stage disagreed — that conflict is the whole point of the override. (4) Document the override file in the pipeline's README as "the tail-end corrector" — otherwise future maintainers will delete it thinking it's dead code. (5) Applies to: ETL pipelines, ML-inference post-processing, address geocoders, product-catalog enrichment, PII redaction. Pair with Lesson 44 (save lessons) and Lesson 286 (don't assume source data) — the override file becomes part of the source data for the next pipeline run.

## MD Merge & Consolidation Workflow (Apr 21, 2026)

476. **Appendix heading prefixes prevent anchor collisions in multi-source MD merges** — When merging multiple MD files verbatim into one document (e.g., `v1.md` + `v2.md` → consolidated `.md`), identical section names across sources ("Objective 1", "Team Roster") collide in the Markdown TOC because both resolve to the same auto-generated anchor slug. Fix: shift original headings down one level AND add a unique per-source prefix such as `A1 Objective 1` / `A2 Objective 1`. Surfaced on 2026-04-21 merging `OKR_KPI_Dashboard_v1.md` + `_v2.md` + `_v8.md` into `OKR & KPI.md` — both v1 and v2 had `## Objective 1` and `## Team Roster`, which without prefixes would produce two `#objective-1` anchors silently clobbering each other. **Why:** Markdown renderers generate anchors from heading slug (`## Team Roster` → `#team-roster`). Two identical H2s in the same document both resolve to the same anchor; the TOC link and any in-document hyperlink jumps only to the first match. Prefixes keep each appendix's navigation independent and audit-ready. **How to apply:** (1) Before merging, pre-scan all source files for duplicate H2/H3 text — `grep -h "^## " *.md | sort | uniq -d`. (2) For every duplicate, demote all copies down one level inside the appendix (so the main doc's `##` section headers still own the top-level TOC) AND add a short per-source prefix (`A1 …` / `A2 …` / `Legacy …`) to each appendix's sub-headings. (3) Apply prefixes only in appendices/supplement content — the primary skeleton (per Lesson #115) keeps its natural headings. (4) Applies to any multi-source consolidation: merged MOM documents, combined quarterly reports, audit-trail archives, multi-version technical specs.

477. **For "merge into one file" tasks, the PM default is full preservation + delete originals** — When asked to consolidate multiple MD/HTML versions into a single target file, two choices compound: **history depth** (full/condensed/current-only) and **source-file fate** (keep/delete/archive). The PM's default workflow is **full preservation** (keep all source tables verbatim in appendices) paired with **delete originals after merge** — the merged file becomes the single audit trail. Confirmed on 2026-04-21 by the OKR & KPI merge (v1 + v2 + v8 → `OKR & KPI.md`). **Why:** Full preservation protects against future "what did v1 actually say" questions; deleting originals prevents fragmented file sprawl and the risk of stakeholders reading outdated v1/v2 as if they were current. The two choices only work together — deleting without preservation loses data, and preservation without deletion defeats the "one file" goal. **How to apply:** (1) Before merging, if the consolidation scope is ambiguous, surface both questions via AskUserQuestion rather than guessing. (2) If the defaults apply, still announce both choices before execution ("I'll preserve v1+v2 as appendices and delete the originals after verification"). (3) Verify the merged file BEFORE any deletion runs: file size ≥ sum of source bodies minus shared boilerplate; grep for unique strings from each source ("Objective 5" → Appendix 1, "Tier 1 — Tracked in Detail" → Appendix 2); confirm all top-level section anchors exist. (4) Only delete after ALL verification checks pass — if any fail, keep the originals and ask the user to inspect. Pair with Lesson #138 (never bash `rm` on Windows user files — for low-risk derivative files after a merge, verified deletion is fine; for originals with irreplaceable content, use Recycle Bin) and Lesson #140 (originals are sacred — once merged, source MDs become derivatives eligible for deletion because the merged file carries 100% of their content).

478. **Plan-mode clarifiers with AskUserQuestion beat guessing on ambiguous consolidation tasks** — Two targeted questions (history depth, source-file fate) turned an ambiguous "merge all this md in one only" request into a deterministic spec. Without them, the output would likely have defaulted to condensed history or left source files in place — either of which would have required a rework cycle. Proven on 2026-04-21 during the OKR & KPI 4-file merge. **Why:** Merge/consolidation requests are structurally ambiguous because the user's mental model rarely distinguishes "consolidate content" from "consolidate files" from "preserve all history" from "keep just current state." Silent guessing produces outputs the user will reject; asking costs 10 seconds and locks the spec before writing a single line. **How to apply:** (1) In plan mode, on any task touching multiple sources/versions, use AskUserQuestion with 2-3 options per question BEFORE finalizing the plan. (2) Put the recommended option first and label it "(Recommended)". (3) Default question set for any merge: (a) history depth — full preservation / condensed / current-only; (b) source-file fate — keep / delete / archive. (4) Skip the clarifiers only if the user pre-specified the choice, or the task is truly trivial (single-line typo fix, one-file rename). (5) Applies beyond merges to any structurally ambiguous request: renaming a folder (move/copy), removing content from a deliverable (delete-from-file/remove-from-publish-only), updating a shared template (in-place/new-version), handing off a project (read-only archive/live-editable).


## Plugin Design — Skills-First vs Hooks-First (Apr 21, 2026)

483. **For Claude Code plugins, authoring-time skills beat post-write hooks for most coverage gaps** — When extending `pm-guidelines-plugin` from v1.4 → v1.7 to close 12 coverage gaps from global_lessons.md, the initial plan leaned heavily on hook extensions (regex checks in `post_write_dispatcher.py`). User course-corrected: "decrease the creation of hooks and use the skills instead as much as possible." The revised plan delivered 3 new skills + 2 new agents + 6 skill extensions + 0 new hooks — closing all 12 gaps with visible authoring-time guidance instead of silent post-write enforcement. **Why:** Hooks (Python scripts on PostToolUse) fire AFTER the mistake lands in the file. They're silent until a regex matches; when they fail they fail quietly; they're Python which carries a cross-platform maintenance tax. Skills (Markdown with prompt-signal activation) fire BEFORE the mistake — at the moment Claude is about to write the wrong thing. They're visible when they activate; they're iterable without restarting the harness; they're domain-portable. The right division of labor: hooks for deterministic security blocks (PAT tokens, protected files), dispatcher for regex-detectable quality checks, skills for everything else (workflows, checklists, judgment-heavy rules, cross-file audits delegated to read-only agents). **How to apply:** (1) When adding new coverage to a plugin, default to a new skill or skill-extension, not a new hook. (2) New hooks are only justified when: the check is regex-detectable AND must fire unprompted (e.g., security gate) AND fits the existing dispatcher's scope. (3) For judgment-heavy or cross-file work, pair an authoring-time skill (prevents) with a read-only audit agent (verifies) — see Lesson #485. (4) Measure coverage via the plugin's own health check after every change; the description-string carries a "N guidelines" pattern that the lesson-count validator expects — keep it current. (5) Applies to any Claude Code plugin, not just pm-guidelines: see `d:\Global Lessons\plans\soft-tinkering-taco.md` for the worked routing table across 12 gaps.

484. **Plan-mode agent forks with different architectural theses produce better plans than a single agent** — When designing the v1.5+ upgrade, I launched TWO Plan agents in parallel with opposing instructions: Agent A was told to maximize new skills/agents; Agent B was told to minimize new files and extend existing surfaces. The final synthesis absorbed the cheap extensions from B's thesis AND the specialist agents from A's thesis — better than either alone. **Why:** A single Plan agent optimizes for its given direction. Two agents with opposing directions surface the actual design space — where one agent's optimum is clearly wrong, the other's is clearly right; where they agree, you have high-confidence structure; where they disagree, you have a genuine architectural fork to present to the user via AskUserQuestion. Cheap (two agents running in parallel is ~same wall time as one) and higher-quality. **How to apply:** (1) For any non-trivial plan involving architectural choices, launch 2-3 Plan agents with EXPLICITLY DIFFERENT theses — "minimize X", "maximize Y", "prefer Z". (2) Brief each agent on the OTHER agent's thesis so they don't accidentally converge. (3) Synthesize by cross-checking divergent recommendations against ground truth (file-system verification, memory entries, CLAUDE.md rules). (4) When both agents arrive at the same recommendation, trust it. When they diverge, offer the user the fork via AskUserQuestion rather than picking silently. (5) Applies to: plugin design, database migration strategy, API redesign, codebase refactoring plans, any task with real architectural trade-offs.

485. **The skill-plus-agent dual-surface pattern is the right shape for judgment-heavy coverage** — Gaps like "dashboard cross-tab number drift" and "MD↔HTML parity" aren't solvable by a single surface. At authoring time, you need a skill that teaches Claude the single-source-of-truth rule BEFORE writing code. At audit time, you need a read-only agent that semantically aligns near-synonym labels ("Open Items" ≈ "Outstanding Tasks") using Opus judgment across the finished file. Using the SAME NAME for the skill and the agent (e.g., `pm-cross-tab-reconciler` skill + `pm-cross-tab-reconciler` agent) reinforces that they solve the same problem at different times — skill prevents, agent verifies. Proven in v1.5+v1.6 of pm-guidelines plugin: cross-tab-reconciler shipped as both skill (Phase 1) and agent (Phase 2); md-html-parity-checker agent (Phase 2) pairs with the `pm-session-discipline` rule "MD is source of truth" (Phase 1). **Why:** Skills activate on prompt signals — they fire while Claude is authoring. Agents are invoked explicitly or via delegation — they fire after authoring is done. Neither alone covers the full lifecycle: a skill with no agent means mistakes that slip past authoring are never caught; an agent with no skill means Claude repeats the mistake every session. The same-name convention makes the pairing visible in the plugin structure (`pm-X/SKILL.md` + `agents/pm-X.md`). **How to apply:** (1) For judgment-heavy gaps, design BOTH surfaces from day one: skill for authoring, agent for audit. (2) Use the same name for both when they solve the same problem; Claude disambiguates via context (prompt signal → skill; explicit launch or reviewer delegation → agent). (3) Cross-reference in both files: skill's body says "if unsure, invoke the agent for post-authoring semantic review"; agent's body says "this is the audit side of the skill — when the skill's discipline is followed, this agent should report PARITY CLEAR". (4) Wire the main reviewer agent to auto-delegate to specialist agents when the target type matches — avoids overloading one agent with every possible check. (5) Applies to any domain with both authoring and audit phases: CI pipeline rules, document quality standards, security review, accessibility compliance.
